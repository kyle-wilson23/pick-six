## What went well

1. **Schema and API consistency** — League, season, memberships, and invitations evolved additively; single `ADMIN` row for creator simplified roster and participant checks for FR12/FR13.
2. **Invitation hardening (2.2)** — Partial unique index for one pending invite per league+email, transactional supersede of stale pending rows, and advisory locking show mature concurrency thinking without scope creep into Epic 6 email.
3. **First competition week as first-class data (2.1 + 2.7)** — Field on `Season`, lock timestamp, admin PATCH with **409** when locked, and participant/rules copy align product and give Epic 3 a clear contract (`firstCompetitionWeek`, `firstCompetitionWeekLockedAt`).
4. **Repeated security baselines** — CSRF/origin checks on mutating routes, structured errors, and admin-only gates matched Epic 1 patterns; delete flow (2.8) used deliberate friction (type `delete`) suitable for irreversible actions.
5. **Colocated tests** — Bodies, helpers, and league rules (e.g. first week, invitations) gained Vitest coverage where logic was pure, keeping feedback fast.

---

## Challenges and friction

1. **Story file status vs sprint-status** — Same class of drift as Epic 1 (e.g. files still showing `review` while sprint shows `done`). Creates noise for anyone using story markdown as source of truth.
2. **Cross-cutting fields** — `firstCompetitionWeek` spanned create flow (2.1), settings (2.7), rules/home copy, and list serializers; required discipline so clients and APIs stayed in sync.
3. **Invite edge cases** — Existing users re-invited still follow generic invalid signup unless product expands “login + accept invite”; documented as follow-up in 2.2.
4. **Rate limiting** — POST paths were the first focus; **`DELETE /api/leagues/[id]`** was later capped in proxy (see Post-retro updates). Other PATCH/DELETE routes may still need explicit limits as Epic 3 adds mutations.
5. **Cascade discipline for delete (2.8)** — Schema documents **`onDelete: Cascade`** for league/season children; Epic 3+ must apply that on every new FK (see Post-retro updates).
