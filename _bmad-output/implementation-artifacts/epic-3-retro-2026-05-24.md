## What went well

1. **Provider spike strategy** — Every external dependency (odds, schedule, weather) was evaluated with documented decision records before implementation. No silent vendor lock-in. The pattern ran 3 times and was clean every time.
2. **Pure domain function architecture** — `lib/domain/jailed.ts`, `pick-deadline.ts`, `picks.ts` all properly separated from the DB layer, all testable in isolation. Jailed logic in particular is a clean reference implementation.
3. **Fixture-based integration testing pattern** — Established in 3.9, followed in 3.10. Deterministic CI, no live network in `npm test`, fixture files as living API documentation.
4. **Fail-soft on all integrations** — Weather missing: silent omit. Odds API down: admin fallback documented. Never cracked across 10 stories. No user-facing crashes from external services.
5. **Mid-implementation course corrections** — Kyle's ability to step in and adjust scope intentionally (dome/retractable roof in 3.10) improved the end result vs. strict spec-following. The code review `decision_needed` gate handled it cleanly.
6. **Cascade FK discipline from Epic 2** — `Pick → Team: Restrict`, `Pick → Season: Cascade` — all new relations documented their `onDelete` strategy. Epic 2 retro commitment carried forward.
7. **`firstCompetitionWeek` wired correctly** — Story 3.1 AC explicitly addresses it; `isWeekInLeagueCompetition` implemented and used throughout. Epic 2 retro commitment fulfilled.

Examples: null `kickoffAt` bypassing guards (3.5), `localeCompare` locale-dependence (3.3), missing test vectors (3.3, 3.5, 3.10).
4. **Thursday 8:10 PM as magic literal** — Deferred in 3.5 review. Still scattered across tests and production code. Not yet a named constant.

## Key insights from deep story analysis

**Patterns across all 10 stories:**

- **Common struggle:** AC-specified edge cases not converted to test stubs during implementation. Appeared in 3.3 (spread sign), 3.5 (Thursday-first-kickoff branch), 3.10 (NaN/past kickoff guards).
- **Recurring review feedback:** Missing boundary guards on null/invalid inputs. Consistently caught by review — consistently not caught during implementation.
- **Breakthrough:** Fixture-based integration test pattern (Story 3.9) — clean, reusable, immediately adopted in 3.10. Worth carrying into any future third-party integration.
- **Team velocity:** Steady across all 10 stories. No significant blockers or rework spirals.
