## Challenges and friction

1. **`AdminPickOverrideDialog.tsx` lint errors â€” six stories, never tracked.** Mentioned as "pre-existing, unrelated" in the completion notes of every story in the epic. Never added to `deferred-work.md`. Story completion notes are write-once artifacts; this item effectively disappeared each time. The structural lesson: "pre-existing, not addressed" in a completion note must also mean "added to `deferred-work.md`."

2. **`$transaction` counter double-counting (Story 5.2).** Counters declared in the outer function scope and mutated inside the `$transaction` callback accumulate across Prisma retry calls. Caught in review and patched; explicitly documented as a Story 5.2 learning for 5.3. The footgun pattern is now in the codebase (both `sync-nfl-results.ts` and `score-nfl-week.ts`).

3. **`ok: false` dead code in result union types (Story 5.2).** The `ScoreNflWeekResult` union includes `{ ok: false }`, but the initial implementation always threw rather than returning it; the route handler's error branch was unreachable. Review caught it. Structural gap: patterns we introduce need to be exercised at all branches.

4. **Test validating mock, not implementation (Story 5.5).** The sort-order test pre-sorted the mock data; deleting `orderBy` from the production query would still pass the test. Caught and patched in review. Tests that validate mock behavior rather than production code create false confidence.

5. **UX spec comparison â€” third retro, still not done.** Originally committed in Epic 3 retro, carried to Epic 4, carried to Epic 5. No sprint-status entry in any of those cycles. Now has one (`6-6-ux-spec-comparison-and-alignment: backlog`).

## Key insights from deep story analysis

**Patterns across all 6 stories:**

- **The deferred-then-addressed pipeline requires the target story to be named in the deferred item.** When it is, the extraction lands cleanly. When it isn't, the item persists indefinitely in completion notes but not in `deferred-work.md`.
- **Review patch count (~4â€“6 per story) has held steady for two epics.** It's not improving. The types of issues (subtle transaction semantics, test quality gaps, unreachable code paths) are harder to pre-empt without a richer implementation checklist.
- **Pattern reuse accelerates and improves quality.** The further into the 5.4â†’5.5â†’5.6 sequence, the fewer review findings â€” because each story author had a working reference implementation to mirror.
- **Privacy enforcement at the query layer was the right call.** `getLeaguePeerPickHistory` makes the gate a single function rather than a scattered set of UI conditions. It's auditable and testable.
