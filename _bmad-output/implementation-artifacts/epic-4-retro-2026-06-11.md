## Challenges and friction

1. **Thursday 8:10 PM magic literal — third epic running.** Epic 3 retro committed to extracting this to a named constant before Epic 5. Epic 4 had no story that touched `pick-deadline.ts`, so the commitment was never executed. The structural lesson: retro commitments without sprint-status entries don't survive.

2. **`validateJailedLineupAndBonus` domain bug — not caught until 4.2 review.** The function returns `JAILED_NOT_IN_WEEK_GAMES` for ALL picks (not just anti-jailed) when the jailed team has a bye. In any bye week during Epic 5, participants cannot submit picks at all. Deferred in 4.2 but elevated to critical path at this retro.

3. **Review volume still ~6 patches/story.** Epic 3 was similar. Domain logic bugs are improving (better test coverage). Client component state bugs (dialog `useEffect` reset, stale `antiJailedBonus`, `setLoading` ordering) are arriving at review rather than being caught during implementation — harder to stub without component testing infrastructure.

4. **Week number skew pattern introduced twice.** In 4.2, `buildSubmissionStatus` and `buildAdminOverrideData` independently called `resolvePicksWeekNumber` in a `Promise.all`, risking week-boundary disagreement. Fixed in 4.2 review, but the same pattern appeared independently in 4.4 (`page.tsx` and `getJailedVerification` each calling the resolver). A structural pattern to address for Epic 5.

## Key insights from deep story analysis

**Patterns across all 4 stories:**

- **Forward-engineering is a repeatable pattern.** When the next story's needs are known at implementation time, designing the current story to accommodate them (passthrough params, optional fields) removes refactoring from the critical path. Do this deliberately in Epic 5 story specs.
- **Parallel fetch with independent week resolvers is a recurring footgun.** Appeared in 4.2 and 4.4 independently. In Epic 5, week transitions are tied to scoring — the stakes for resolver disagreement are higher.
- **Client component state bugs are hard to pre-empt without infrastructure.** Dialog lifecycle edge cases (open/close resets, async loading state ordering) aren't testable with current unit test setup. May be worth evaluating component testing for Epic 5's participant-facing views.
- **Deferred items from prior epics are best tracked by naming them in the target epic's story spec.** The `adminMembershipId` passthrough and `afterMoneyline` slices were both deferred from earlier epics and landed cleanly because the target story named them explicitly.
