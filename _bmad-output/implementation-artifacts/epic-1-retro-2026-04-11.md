## What went well

1. **Clear sequencing** — Foundation, data, auth, session tuning, invites, and guards followed the epic plan without scope drift (JWT + Credentials stayed consistent through 1.6).
2. **Adversarial code review paid off** — Repeated passes caught theme consistency (1.1 gold accent), tooling (ESLint + CJS scripts, `dotenv` in `dependencies`), security hardening (constant-time bcrypt path in 1.3), framework churn (Next 16 `proxy` vs `middleware`), signup recovery when `signIn` fails after user creation (1.5), and CSRF helper naming + negative tests (1.6).
3. **Colocated tests for pure logic** — `session-constants`, invitations helpers, `callback-url`, and CSRF helpers gained Vitest coverage, matching project norms and keeping feedback fast.
4. **Single Prisma client + documented Neon pattern** — Singleton `db.ts`, `DIRECT_URL` for migrations, and additive migrations through auth adapter and invitations reduced integration risk.
5. **Explicit security baselines** — Rate limiting on credential and signup POST paths, safe `callbackUrl` handling, invite token hashing, and generic error surfaces for anti-enumeration aligned with NFRs called out in stories.

---

## Challenges and friction

1. **Auth.js v5 beta + Next 16** — Catch-all route naming, JWT vs DB session semantics (`updateAge` documentation), and `proxy` migration required careful doc alignment and a few review cycles.
2. **Rate-limit key strategy** — Iterated from shared `unknown` bucket to IP + UA-hash fallback (1.5); still worth validating under real proxies and multi-instance deployment.
3. **Signup complexity** — Transactional user + invite consume + client `signIn` created edge cases (post-create sign-in failure); remediation improved UX but increased surface area.
4. **Proxy vs layout coupling** — Protected layout relies on `x-pathname` from `proxy`; matcher must stay in sync when new protected paths are added under `(app)` (called out in 1.6 review).
5. **Artifact hygiene** — Story `1-5` markdown still shows `Status: review` while sprint-status lists `done`; reconcile file headers with sprint to avoid confusion.
